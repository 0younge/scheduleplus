# 일정 관리 API

일정(Schedule), 유저(User), 댓글(Comment) CRUD 기능을 제공하는 RESTful API 서버입니다.

---

## 기술 스택

- Java / Spring Boot
- Spring MVC
- JPA / Hibernate
- MySQL
- Session 기반 인증

---

## 필수 기능

### Lv 0. API 명세 및 ERD 작성

- [x] API 명세서 작성 (프로젝트 root 경로의 `README.md`에 작성)
- [x] ERD 작성 및 첨부 (프로젝트 root 경로의 `README.md`에 첨부)


### Lv 1. 일정 CRUD

- [x] 일정 생성 / 전체 조회 / 단건 조회 / 수정 / 삭제
- [x] 일정 필드
  - `작성 유저명`, `할일 제목`, `할일 내용`, `작성일`, `수정일`
  - `작성일`, `수정일`은 **JPA Auditing** 활용


### Lv 2. 유저 CRUD

- [x] 유저 생성 / 전체 조회 / 단건 조회 / 수정 / 삭제
- [x] 유저 필드
  - `유저명`, `이메일`, `작성일`, `수정일`
  - `작성일`, `수정일`은 **JPA Auditing** 활용
- [x] 연관관계 구현
  - 일정의 `작성 유저명` 필드 → `유저 고유 식별자` 필드로 변경


### Lv 3. 회원가입

- [x] 유저에 `비밀번호` 필드 추가
  - 비밀번호는 **8글자 이상** 필수
  - 비밀번호 암호화는 Lv 6(도전 기능)에서 구현


### Lv 4. 로그인 (인증)

- [x] **Cookie / Session** 방식으로 로그인 기능 구현
- [x] `이메일`과 `비밀번호`를 활용한 로그인
- [x] 필요한 API에서 세션 활용

## 도전 기능

### Lv 5. 다양한 예외처리

- [x] `@Validation`을 활용한 예외처리 적용
- [x] `@RestControllerAdvice`로 validation 에러를 클라이언트에 전달
- [x] 프로젝트 분석을 통한 예외 항목 지정
  - 예시: 할일 제목 10글자 이내, 유저명 4글자 이내


### Lv 6. 비밀번호 암호화

- [x] Lv 3에서 추가한 `비밀번호` 필드 암호화 적용
- [x] 암호화를 위한 **`PasswordEncoder` 직접 구현**


### Lv 7. 댓글 CRUD

- [x] 일정에 댓글 작성 기능 구현
- [x] 댓글-일정 연관관계 매핑
- [x] 댓글 저장 / 전체 조회
- [x] 댓글 필드
  - `댓글 내용`, `작성일`, `수정일`, `유저 고유 식별자`, `일정 고유 식별자`
  - `작성일`, `수정일`은 **JPA Auditing** 활용


### Lv 8. 일정 페이징 조회

- [x] Spring Data JPA의 `Pageable`, `Page` 인터페이스를 활용한 페이지네이션
  - `페이지 번호`, `페이지 크기`를 쿼리 파라미터로 전달
  - 디폴트 페이지 크기: **10**
- [x] 조회 필드: `할일 제목`, `할일 내용`, `댓글 개수`, `일정 작성일`, `일정 수정일`, `일정 작성 유저명`
- [x] `수정일` 기준 **내림차순** 정렬

## 트러블 슈팅

### 1. Properties 설정 노출
- **문제**: 환경 설정 정보를 가리지 못한 채 Git에 커밋됨
- **해결**: 이후 환경변수로 수정했으나, 이미 깃에 올라간 이후라 아쉬움이 남음
- **개선점**: 개발 초기부터 `.gitignore` 및 환경변수를 철저히 관리


### 2. 중복 코드 파편화
- **문제**: 유사한 로직이 여러 곳에 중복 작성되어 코드 파편화 발생
- **해결**: 중복 로직을 별도 메서드로 추출하여 재사용
- **개선점**: 처음 설계 단계에서 공통 로직을 미리 분리


### 3. 세션 구현 방식 이해 부족
- **문제**: Cookie/Session 기반 인증 구현 방식이 처음에 이해되지 않음
- **해결**: 튜터 질문 및 검색을 통해 작성 → 이후 스터디에서 다른 수강생에게 직접 설명하며 완전히 이해
- **개선점**: 단순히 코드 작성법이 아닌 동작 원리부터 이해하는 습관 형성


### 4. 예외 클래스 미분리
- **문제**: 커스텀 예외를 만들지 않고 기존 예외를 그대로 활용
  - 특정 예외가 어디서 발생하는지 추적이 어려움
  - 오류가 자주 발생하는 지점을 파악할 수 없게 됨
- **해결**: 해당 과제에서는 기존 예외로 처리
- **개선점**: 다음 프로젝트부터는 커스텀 예외 클래스를 직접 만들어 사용처 추적 및 오류 통계 관리

## Auth패키지의 의미

user패키지와 같은 엔티티를 공유하며 URL도 동일합니다

- user에서 처리하는게 맞다고 생각하지만 기능단위로 분류하고 싶어 패키지를 분리했습니다

## common패키지의 의미

공통적으로 사용하거나 전체 패키지에서 사용되는 클래스들을 모아두었습니다

- BaseController: 세션 여부를 확인하는 메서드가 전체 컨트롤러에서 중복되어 추상화를 통해 구현하였습니다
